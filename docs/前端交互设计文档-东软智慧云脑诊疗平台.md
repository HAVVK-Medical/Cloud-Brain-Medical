# 东软智慧云脑诊疗平台前端交互设计文档

**版本**：v1.0  
**日期**：2026-06-24  
**依据**：[`PRD-东软智慧云脑诊疗平台.md`](./PRD-东软智慧云脑诊疗平台.md)、[`系统设计文档-东软智慧云脑诊疗平台.md`](./系统设计文档-东软智慧云脑诊疗平台.md)、[`技术设计.md`](../designs-tech/202606221721_cloud-brain-medical-tech-design/tech_v1.md)

## 1. 文档目标

本文档定义前端三端界面的交互结构、页面层级、导航方式、状态反馈、响应式规则和组件拆分原则，作为后续前端重构的统一标准。

核心目标：

1. 让患者端、医生端、管理端各自只看到自己的主任务。
2. 将“分诊-挂号-接诊-病历-处方-审核-反馈”做成清晰的闭环。
3. 让移动端适配患者，PC 端适配医生和管理后台。
4. 让实时通知、SSE 流式结果和数据看板有统一的交互出口。

## 2. 设计原则

1. 三端分离，入口统一。
2. 信息分层，先任务后数据。
3. 同屏密度适中，减少单页堆叠。
4. 状态必须可见，加载、错误、降级都要有明确反馈。
5. 手机端优先简洁操作，PC 端优先并行查看和编辑。
6. AI 能力作为辅助层，不抢占主流程。

## 3. 角色与终端

| 角色 | 终端形态 | 主任务 |
|---|---|---|
| 患者 | 手机端优先 | 分诊、挂号、看病历、看处方、提交反馈 |
| 医生 | PC 工作台 | 接诊、问诊、病历生成、处方审核、完成就诊 |
| 管理员 | PC 后台 | 基础数据维护、排班、药品、规则、AI 配置、审计看板 |

## 4. 全局信息架构

### 4.1 入口层

- `/`：系统入口与角色导流。
- `/login`：统一登录/注册入口。
- `/patient/**`：患者工作区。
- `/doctor/**`：医生工作区。
- `/admin/**`：管理工作区。

### 4.2 全局导航

- 顶部保留系统品牌、当前角色、健康状态、刷新入口、退出入口。
- 角色工作区内部使用二级导航，不把所有功能平铺在一个页面里。
- 患者端优先底部/顶部轻量标签切换。
- 医生端和管理端优先左侧工作台导航 + 右侧编辑区。

## 5. 患者端交互

### 5.1 页面结构

患者端按“概览 / 分诊 / 挂号 / 病历 / 资料 / 历史”组织。

- 概览：当前分诊结果、当前挂号、待办数、最近更新时间。
- 分诊：输入主诉，展示推荐科室、医生、号源。
- 挂号：选择科室、医生、时段，确认挂号。
- 病历：查看就诊记录、处方、审核结果。
- 资料：维护个人信息、联系方式、既往史。
- 历史：查看历史分诊、历史挂号、反馈记录。

### 5.2 关键交互

1. 用户先输入主诉。
2. 系统触发本地规则分诊，必要时显示 AI 建议结果。
3. 用户在推荐科室和可用号源中完成选择。
4. 用户确认挂号后，系统回写挂号状态。
5. 就诊后可查看病历、处方和风险提示。
6. 完成后可提交反馈，形成质量闭环。

### 5.3 手机端规则

- 采用单列布局。
- 标签页横向可滚动。
- 表单控件默认 16px，避免移动端自动缩放。
- 主要按钮全宽显示。
- 列表项采用“标题 + 状态 + 辅助信息”三层结构。

## 6. 医生端交互

### 6.1 页面结构

医生端按“总览 / 接诊 / 历史 / 排班”组织。

- 总览：今日接诊、待处理队列、风险提示、消息提醒。
- 接诊：问诊文本、病历草稿、诊断建议、处方编辑、处方审核、提交就诊。
- 历史：既往病历、处方历史、审方历史。
- 排班：查看自己的排班和号源。

### 6.2 关键交互

1. 进入工作台后先看到待接诊队列。
2. 选择患者后加载工作区快照。
3. 问诊文本可触发 SSE 流式输出。
4. 病历草稿可由 AI 辅助生成，再由医生确认。
5. 处方先本地规则审核，再展示模型解释。
6. 提交处方与结束就诊都要二次确认。

### 6.3 工作台规则

- 左侧固定队列与导航，右侧主编辑区。
- 处方、病历、审核结果分区展示，避免一屏堆叠。
- 处方风险和规则命中必须高可见。
- WebSocket 通知单独挂载，不占主工作区。

## 7. 管理端交互

### 7.1 页面结构

管理端按“总览 / 基础数据 / 排班与资源 / 配置 / 审计”组织。

- 总览：核心指标、风险趋势、AI 调用统计、反馈统计。
- 基础数据：科室、医生、药品。
- 排班与资源：号源、时段、剩余量。
- 配置：审方规则、AI 配置、Prompt 模板。
- 审计：AI 调用记录、审计日志、未读告警。

### 7.2 关键交互

1. 总览页用于判断系统状态。
2. 资源页用于检索、编辑、新增、启停。
3. 配置页用于规则版本与 AI 提供方配置。
4. 审计页用于追溯风险和操作链路。

### 7.3 后台规则

- 表格与表单分区，列表不直接淹没编辑区。
- 重要配置必须可追溯。
- AI 配置仅展示脱敏信息，不展示明文密钥。

## 8. 状态与反馈

### 8.1 加载

- 页面首屏使用骨架或占位块。
- 局部刷新使用按钮 loading。
- SSE / WebSocket 连接状态要可见。

### 8.2 空状态

- 列表为空时显示明确业务提示，而不是空白区。
- 无号源、无通知、无历史记录都要区分展示。

### 8.3 错误与降级

- 网络错误显示中文提示。
- AI 不可用时保留本地规则流程。
- 实时推送失败不阻塞主流程。

### 8.4 反馈原则

- 成功要有确认。
- 关键操作必须二次确认。
- 高风险结果必须突出显示。

## 9. 组件拆分建议

| 层级 | 建议组件 |
|---|---|
| 应用壳 | `AppShell`、`RoleWorkspaceFrame` |
| 通用区 | `StatusChip`、`SectionCard`、`EmptyState`、`ConfirmDialog` |
| 患者端 | `PatientOverviewPanel`、`PatientTriagePanel`、`PatientRegistrationPanel`、`PatientRecordPanel`、`PatientProfilePanel`、`PatientHistoryPanel` |
| 医生端 | `DoctorOverviewPanel`、`DoctorConsultationPanel`、`DoctorHistoryPanel`、`DoctorSchedulePanel` |
| 管理端 | `AdminOverviewPanel`、`AdminMasterDataPanel`、`AdminConfigPanel`、`AdminAuditPanel` |

## 10. 路由与状态

### 10.1 路由

- 保持 `/patient/`、`/doctor/`、`/admin/` 三端分离。
- 工作区内部建议继续细分子路由，避免单页承担全部状态。

### 10.2 状态

Pinia Store 继续按角色和业务域拆分：

- `useAuthStore`
- `usePatientStore`
- `useDoctorStore`
- `useAdminStore`
- `useRegistrationStore`
- `useScheduleStore`
- `usePrescriptionStore`
- `useMedicalRecordStore`
- `useDrugStore`
- `useDashboardStore`
- `useNotificationStore`
- `useAIStore`

## 11. 验收标准

1. 患者端在手机宽度下可顺畅完成分诊和挂号。
2. 医生端能在单个工作区内完成接诊、病历、处方、提交。
3. 管理端能维护科室、医生、排班、药品、规则和 AI 配置。
4. SSE、WebSocket、看板和错误态都有明确反馈。
5. 页面结构清晰，模块边界可见，不再出现所有业务堆在单页的情况。
