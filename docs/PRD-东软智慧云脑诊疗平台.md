## 1. 产品概述与愿景

### 1.1 产品定位

东软智慧云脑诊疗平台是一套面向高校教学实训的**全流程智能诊疗系统**，采用 Spring Boot 3 + Vue 3 前后端分离架构，融合 AI 大模型、知识图谱与大数据分析技术，覆盖"诊前—诊中—诊后"完整医疗闭环。

平台定位为**教学实训产品**，目标不是生产级医疗系统，而是让全栈开发学习者通过动手实践，理解传统业务模块如何与 AI 能力有机结合、前后端如何通过 RESTful API 协作。

### 1.2 产品愿景

让每一位学生通过本项目，完整理解以下核心命题：

- **前后端分离架构**下，Spring Boot 3（后端）与 Vue 3（前端）分别承担什么职责，如何通过 RESTful API + JWT 认证协作。
- **传统医疗业务**（患者管理、药品管理、挂号、处方、病历）如何从数据库设计到前端交互的完整链路。
- **AI 能力融合**：智能分诊、AI 辅助诊断、AI 处方审核、AI 病历生成如何作为独立 Service 嵌入传统业务流程，分别扮演"智能入口""辅助决策""副驾驶""人机协作"角色。
- **数据驱动闭环**：通过就诊评价与反馈数据、数据可视化看板，体现"患者需求感知—诊疗质量反馈"的大数据分析闭环。
- **全栈联调**：如何处理跨域、异步状态、错误边界、AI 服务降级等真实工程问题。

### 1.3 核心价值主张

| 维度 | 价值 |
|------|------|
| **患者端** | 输入症状 → AI 推荐科室医生 → 一键挂号 → 查看处方病历 → 就诊后评价反馈，全流程在线化闭环 |
| **医生端** | 查看患者 → 录入对话 → AI 辅助诊断 → AI 生成病历 → 选择药品 → AI 审核处方 → 安全开方 → 数据看板，AI 赋能全流程 |
| **学习者** | 完整的前后端分离 + AI 融合 + 大数据分析开发训练，从零到一交付可运行的全栈应用 |

## 4. 患者端功能需求

### 4.1 注册与登录

**【功能概述】**

患者通过注册创建账号，登录后获取 JWT 令牌，后续请求自动携带令牌以访问受保护接口。

**【功能要点】**

| ID | 需求描述 |
|----|---------|
| P-REG-01 | 注册页面包含：用户名、手机号、密码、确认密码等字段 |
| P-REG-02 | 前端对密码长度（≥6 位）、手机号格式进行校验 |
| P-REG-03 | 注册成功后跳转至登录页 |
| P-REG-04 | 登录页面包含：用户名/手机号、密码 |
| P-REG-05 | 登录成功后后端返回 JWT 令牌，前端存储于 localStorage |
| P-REG-06 | Axios 拦截器自动在请求头携带 token：`Authorization: Bearer <token>` |
| P-REG-07 | 登录失败时页面提示错误原因（密码错误/用户不存在） |

**【数据模型关联】**

- 患者表（Patient）：存储患者账号与基本信息

### 4.3 智能分诊与挂号

**【功能概述】**

患者输入症状描述后，系统调用 AI 分诊引擎，推荐最合适的科室及医生列表。患者可在推荐结果中直接选择医生完成挂号。

这是 **AI 与传统业务融合的核心场景之一**——AI 作为挂号的智能入口。

**【功能要点】**

| ID | 需求描述 |
|----|---------|
| P-TRI-01 | 智能分诊页面提供多行文本输入框，供患者输入症状描述（如"胸痛伴气短"） |
| P-TRI-02 | 点击"智能分诊"按钮后，调用后端 AI 分诊接口 |
| P-TRI-03 | 调用期间显示 Loading 加载状态（骨架屏或旋转动画），按钮置灰不可重复点击 |
| P-TRI-04 | AI 返回推荐结果后，以卡片列表形式展示推荐科室及科室下的推荐医生 |
| P-TRI-05 | 每张医生卡片展示：医生姓名、所属科室、职称、简介 |
| P-TRI-06 | 医生卡片上提供"立即挂号"按钮，点击后跳转至该医生的挂号页面 |
| P-TRI-07 | 挂号页面展示医生信息、可选时间段，患者选择时间段后提交 |
| P-TRI-08 | 挂号成功后弹出成功提示，自动跳转至"我的挂号"页面 |
| P-TRI-09 | AI 调用失败时，显示"AI 分诊暂时不可用，请手动选择科室"降级提示，并提供手动科室筛选入口 |

**【用户操作流程】**

```
输入症状 → 点击"智能分诊" → [Loading] → 查看推荐科室与医生
    ├─ 点击"立即挂号" → 选择时间段 → 确认挂号 → 成功提示
    └─ AI 不可用 → 提示降级 → 手动浏览科室 → 选择医生 → 挂号
```

### 4.7 就诊评价与反馈

**【功能概述】**

患者就诊完成后，可对本次就诊服务进行评价，并对 AI 分诊准确度进行反馈。评价数据持久化入库，用于后续 AI 推荐优化与数据看板展示。该模块实现了课程要求的 **"患者需求感知—诊疗质量反馈"闭环**。

**【功能要点】**

| ID | 需求描述 |
|----|---------|
| P-FB-01 | 挂号记录状态变为"已完成"后，显示"评价"入口按钮 |
| P-FB-02 | 评价页面包含：服务评分（1-5 星）、AI 分诊准确度反馈（"推荐科室是否正确？"是/否）、文字评价（选填） |
| P-FB-03 | 提交评价后反馈"评价成功"，评价记录持久化入库 |
| P-FB-04 | 历史评价可在个人中心或挂号详情中查看 |

### 6.5 AI 能力降级策略

AI 服务为外部依赖，存在网络延迟、超时、服务不可用等风险。平台需内置降级策略：

| 场景 | 降级行为 |
|------|---------|
| AI 分诊不可用 | 前端提示"AI 分诊暂时不可用"，展示手动科室选择入口，不阻塞挂号 |
| AI 辅助诊断不可用 | 前端提示"AI 诊断暂不可用"，医生自行判断诊断方向 |
| AI 处方审核不可用 | 前端提示"AI 审核暂不可用"，医生可直接提交处方 |
| AI 病历生成不可用 | 前端提示"AI 生成暂不可用"，医生可直接手动填写病历表单 |
| AI 调用超时 | 设置合理超时时间（建议 15-30 秒），超时后走上述降级路径 |
| AI 返回格式异常 | 后端解析失败时返回降级响应，前端按不可用处理 |

**核心原则**：AI 是辅助，不应成为流程瓶颈。AI 不可用时系统核心流程（挂号、诊断、开方、保存病历）必须保持可用。
